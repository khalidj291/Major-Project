# Data Documentation — Memory That Fades

Written by Person 1 (data pipeline owner). Covers where the data came from,
how it was cleaned, how regime labels are computed, and the known
limitations/issues found and fixed during the project.

---

## 1. Data Sources

### Financial domain
- **Source:** Yahoo Finance, via the `yfinance` Python library
- **Tickers pulled:** AAPL, SPY, BTC-USD
- **Date range:** 2015-01-01 to 2024-12-31
- **Frequency:** Daily
- **Fields pulled:** closing price, trading volume
- **Primary asset used for modeling:** SPY (S&P 500 ETF) — chosen for
  liquidity and comparability to existing literature. AAPL and BTC-USD are
  present in the raw data but not currently used in the trained models
  (BTC-USD is parked as a possible stretch-goal domain, not a core one).

### Consumer domain
- **Source:** FRED (Federal Reserve Economic Data), series ID `PCE`
  (Personal Consumption Expenditures)
- **Date range:** 2015-01-01 to 2024-12-31
- **Frequency:** Monthly
- **Fields pulled:** PCE level, mapped to `close` in our schema; `volume`
  set to a 1.0 placeholder since PCE has no volume concept

## 2. Cleaning and Feature Engineering

- `returns` calculated as the day-over-day (or month-over-month for
  consumer) percentage change in `close`: `(today - yesterday) / yesterday`
- First row of each ticker/series dropped (no prior value to compute a
  return from)
- Rows with null `close` or `volume` removed
- **Windowing convention (financial and consumer both):** for day/month
  `i`, the feature vector is the previous `window` returns, `returns[i-window:i]`,
  and the prediction target is `returns[i]` — the same day/month
  immediately following the window, no gap. Window = 30 (days) for
  financial, window = 12 (months) for consumer, since 30 months would
  consume most of the consumer dataset's history.
- **Train/test split (both domains):** train on 2015-01-01 to 2022-12-31,
  test on 2023-01-01 to 2024-12-31. This boundary is used identically
  across every training and evaluation script in the repo and should not
  be changed without updating it everywhere at once.

## 3. Regime Labeling

- Computed on SPY only, from `baseline_model/add_regime_labels.py`
- Method: 30-day rolling standard deviation of `returns`
- Top 30% of rolling-volatility values → labeled `volatile`
- Bottom 30% → labeled `stable`
- Middle 40% → labeled `neutral`
- Output: `regime`/`rolling_vol` columns added to the SPY rows of
  `data/data_processed.csv`, plus a standalone `data/data_regime.csv`
  (SPY-only, 2,514 rows, ticker column dropped)
- Sanity-checked against known volatile events (COVID crash, 2022 rate
  hikes) — both land inside "volatile" as expected
- Current test-period (2023-2024) regime split: 62 volatile days,
  96 stable days, out of 501 total test days

## 4. Known Issues Found and Fixed

### Ticker-mixing bug (fixed)
`data_processed.csv` is intentionally multi-ticker (AAPL/BTC-USD/SPY
stacked, 8,679 rows total) by design. `generate_weights.py` and
`train_decay_models.py` were initially missing a ticker filter, meaning
decay weights and models were computed over interleaved rows from three
unrelated assets. Fixed by adding an explicit `ticker == "SPY"` filter to
both scripts.

### Windowing target offset bug (fixed)
The decay-side windowing function originally set the prediction target to
`returns[i+1]` while the baseline used `returns[i]` for the same feature
window `returns[i-window:i]` — a one-day gap with no modeling
justification, present in 7 files (`train_decay_models.py` and its
consumer variant, `decay_rate_sensitivity.py`, both regime analysis
scripts, and both statistical significance scripts). This meant baseline
and decay models were technically being evaluated on slightly different
prediction tasks. Fixed by aligning all 7 files to the baseline's
convention (target = `returns[i]`, no gap). All models were retrained
after this fix.

### `add_regime_labels.py` default-ticker bug (fixed)
The function originally defaulted to `ticker=None`, which fell back to
treating the entire (multi-ticker) `data_processed.csv` as if it were a
single asset when called without an explicit ticker — the same class of
bug as the ticker-mixing issue above, in a different script. Fixed by
requiring an explicit ticker argument (now defaults to `"SPY"`) and
removing the no-filter fallback path.

### Redundant/stale files
`data/data_volatile.csv`, `data/data_stable.csv`, `data/data_neutral.csv`
were generated by `split_by_regime.py` before the ticker-mixing fix and
still contain all three tickers mixed together. No current script
consumes them; they are not used anywhere in the pipeline and can be
regenerated or deleted. `data/data_regime.csv` is the correct, current,
SPY-only regime file actually used by the regime-analysis scripts.

## 5. Sample Size Limitations

- Consumer domain: only ~120 total monthly rows (2015-2024), with just
  24 rows in the held-out test period and as few as 9-12 rows in any
  single regime subset. Any consumer-domain regime-specific result should
  be read as directional at best, not statistically reliable at this
  sample size.
- Financial domain: 62 volatile test days out of 501 total — larger than
  consumer, but still underpowered for detecting small effect sizes.
  Extending the financial pull further back than 2015 (SPY has data to
  1993) would meaningfully increase the volatile-day sample if pursued.
- BTC-USD: parked as a stretch goal, not part of the core reported
  results. Its own historical volatility events (2018, 2020, 2022) mostly
  fall before the shared 2023-2024 test window, so a BTC-specific
  train/test split would likely be needed if pursued, rather than reusing
  the SPY/PCE boundary.
